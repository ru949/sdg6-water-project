4. Reflection on the Validation Process

HTML validation was performed on each page after the main implementation was complete, using the W3C Markup Validation Service via the "Validate by Direct Input" method.

Errors Encountered and Resolved

During initial validation, a small number of errors were found and corrected before the final screenshots were taken. These included:

Error 1 Unclosed element: An early draft of feedback.html had a <div> inside the programme grid that was not closed correctly. The validator flagged this immediately. It was corrected by ensuring all <div> elements had matching closing tags, and re-running validation to confirm.

Error 2 Missing alt attribute: During team.html development, one of the team member <img> elements was initially missing its alt attribute. The validator flagged this as an error. All images were then checked systematically and descriptive alt text was added to every image across all three pages.

Error 3 Invalid nesting: In content_ST3.html, a <p> element was accidentally nested inside another <p> element (a common HTML error). The validator detected this and it was corrected by closing the outer paragraph before opening the nested content in a <div> instead.

Lessons Learned

Running validation regularly during development —rather than only at the end would have caught these errors earlier and reduced rework. The W3C validator provides clear, line-numbered error messages that make debugging straightforward. Going forward, I will validate after completing each major section of a page rather than waiting until the entire page is finished.
